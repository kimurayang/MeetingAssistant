## 1. 簡介

### 1.1 文件目的
本文件旨在詳細定義「離線 AI 會議助手」最小可行產品 (MVP) 的功能需求、技術架構、用戶體驗以及成功指標。此文件將作為開發團隊、設計團隊及利害關係人之間的溝通基準，確保產品開發符合業務目標並解決用戶痛點。

### 1.2 產品背景
在現代企業環境中，會議紀錄是資訊傳遞與決策追蹤的核心。然而，手寫紀錄效率低下且容易遺漏細節。現有的雲端語音轉文字解決方案（如 Otter.ai 或 Zoom AI Companion）雖然強大，但對於軍事、醫療、金融及高科技研發等對**資料隱私**有極高要求的產業而言，將敏感的會議音訊上傳至第三方雲端伺服器存在巨大的資安風險。市場上迫切需要一款能在本地環境運行、不需聯網且具備高品質轉錄與 AI 摘要能力的工具。

### 1.3 產品願景
打造全球最安全、最智能的本地端會議決策支援系統。讓用戶在享受 AI 技術帶來的便利時，無需擔心任何數據外洩，實現 100% 的數位主權與隱私保護。

### 1.4 MVP 目標
*   **本地音訊擷取**：實現穩定且高音質的電腦內建/外接麥克風錄音。
*   **全離線語音轉文字 (STT)**：整合 Whisper 模型，在無網路環境下達成 90% 以上的轉錄準確度。
*   **自動發言人識別 (Diarization)**：準確區分並標註 2-5 人的發言段落。
*   **本地 AI 語意分析**：利用本地 LLM 生成包含決策點與待辦事項的會議摘要。
*   **高效紀錄管理**：提供完整的歷史紀錄檢索與編輯功能。
*   **多格式數據輸出**：支援將產出結果匯出為通用辦公格式（PDF/TXT）。

### 2. 目標用戶

*   **主要用戶**：
    *   **企業高階主管/秘書**：需要準確紀錄決策過程，並快速產出會議摘要。
    *   **研發工程師/專案經理**：在討論技術架構或專案進度時，紀錄技術細節與 Action Items。
    *   **法律與醫療從業者**：對訪談或診斷過程進行高度隱私的紀錄。
*   **用戶特徵**：
    *   **硬體設備優良**：通常配備中高階商務筆電或工作站。
    *   **隱私敏感度極高**：不信任雲端存儲，甚至在物理隔絕 (Air-gapped) 的環境中工作。
    *   **效率導向**：希望減少會後整理紀錄的時間，直接獲得結構化資訊。

---

### 3. 核心問題與解決方案

*   **核心問題 1：資料隱私與安全性**
    *   描述：企業會議內容涉及商業機密，傳統雲端轉錄工具存在數據被攔截或被供應商用於模型訓練的風險。
    *   **MVP 解決方案**：採用「Local-First」架構，所有音訊處理、轉錄與 AI 推理均在使用者本機執行，物理上切斷數據上傳路徑。
*   **核心問題 2：多人口說混亂**
    *   描述：會議紀錄最難的是區分「誰在什麼時候說了什麼」，單純的逐字稿難以閱讀。
    *   **MVP 解決方案**：整合 `pyannote-audio` 離線模型進行聲紋識別，自動標註「發言人 1」、「發言人 2」，並與時間戳對齊。
*   **核心問題 3：逐字稿過長，難以抓到重點**
    *   描述：長達一小時的會議會產生上萬字的逐字稿，使用者缺乏時間閱讀全文。
    *   **MVP 解決方案**：整合本地輕量化大語言模型 (LLM)，自動提取會議主旨、決策結論及待辦清單 (Action Items)。
*   **核心問題 4：歷史資料搜尋困難**
    *   描述：錄音檔案散落在資料夾中，難以根據關鍵字找到特定會議的特定對話。
    *   **MVP 解決方案**：建立本地 SQLite 資料庫，針對所有轉錄文字建立索引，提供全文檢索功能。

## 4. 產品功能 (MVP 範圍)

### 4.1 錄音與音訊控制模組
*   **功能描述**：負責高品質音訊的即時擷取與儲存。
*   **具體子功能**：
    *   **設備選取**：自動偵測並允許用戶切換輸入源（內建麥克風、USB 麥克風、藍牙耳機）。
    *   **即時音量波形**：以可視化動態波形顯示音量，確保錄音設備正常作動。
    *   **錄音控制**：提供開始、暫停、繼續、停止錄音功能，並即時顯示已錄製時長。
    *   **本地緩存**：錄音過程中自動分段寫入暫存檔，防止程式崩潰導致資料全失。

### 4.2 離線語音轉文字 (STT) 引擎
*   **功能描述**：將音訊檔案轉換為帶有時間戳的文字。
*   **具體子功能**：
    *   **模型選擇**：提供不同規模的 Whisper 模型 (Tiny/Base/Small/Medium) 供用戶根據硬體效能選用。
    *   **多語言偵測**：自動識別會議語言（支援繁體中文、英文、日文等）。
    *   **時間戳對齊**：每一段轉錄文字皆需精確對應音訊時間（Start/End Time）。
    *   **專有名詞優化**：支援用戶導入自定義詞典（如公司名、產品名），提升識別準確率。

### 4.3 發言人辨識模組 (Speaker Diarization)
*   **功能描述**：分析音訊聲紋，區分不同說話者。
*   **具體子功能**：
    *   **聲紋特徵提取**：自動分析音訊中的音色特徵。
    *   **說話者聚類**：將相同特徵的音段歸類為同一位發言人。
    *   **發言人重新命名**：用戶可在轉錄完成後，將「發言人 1」手動修改為真實姓名（如「王大明」），系統自動更新全篇標註。

### 4.4 AI 智能摘要與分析模組
*   **功能描述**：利用本地 LLM 對轉錄內容進行結構化分析。
*   **具體子功能**：
    *   **自動摘要**：生成 300-500 字的會議簡述。
    *   **決策提取**：自動列出會議中達成的共識與決策事項。
    *   **行動清單 (Action Items)**：識別會議中指派的任務、負責人及截止日期。
    *   **關鍵詞提取**：自動標註會議中頻繁出現的核心詞彙。

### 4.5 會議紀錄管理中心
*   **功能描述**：管理歷史會議資料。
*   **具體子功能**：
    *   **列表檢視**：以列表形式展示歷史會議，包含標題、日期、時長、參與人數。
    *   **逐字稿編輯器**：支援手動修正轉錄錯誤，並即時儲存。
    *   **全文搜索**：支援透過關鍵字搜尋所有歷史紀錄中的對話內容。
    *   **錄音回放**：點擊特定文字段落，自動跳轉並播放對應的錄音片段。

### 4.6 數據匯出系統
*   **功能描述**：將處理結果轉換為可分享的檔案。
*   **具體子功能**：
    *   **PDF 匯出**：生成包含摘要、發言人標註、詳細紀錄的格式化 PDF 報告。
    *   **純文字匯出**：支援 TXT 或 Markdown 格式，方便用戶複製到其他工具。
    *   **音訊匯出**：支援將原始錄音檔另存為 WAV 或 MP3 格式。

## 5. 用戶故事 (User Stories)

1.  **作為** 會議負責人，**我想要** 在離線環境下直接錄製實體會議，**以便** 確保高度機密的研發會議不會被上傳到雲端。
2.  **作為** 專案經理，**我想要** 系統自動區分不同的發言者，**以便** 整理會議紀錄時能快速確認誰對某項提議表示反對。
3.  **作為** 行政秘書，**我想要** AI 自動生成會議摘要和待辦事項，**以便** 我能在會議結束後 5 分鐘內將紀錄發送給所有成員。
4.  **作為** 法務人員，**我想要** 在逐字稿中搜尋關鍵字，**以便** 快速定位證人或當事人在長達 3 小時訪談中的特定發言。
5.  **作為** 軟體工程師，**我想要** 自定義專業術語詞庫，**以便** 提高 AI 對於「K8s」、「Microservices」等特定名詞的識別準確度。
6.  **作為** 一般職員，**我想要** 點擊逐字稿中的某一行就能播放那一段的錄音，**以便** 確認 AI 是否有誤解當事人的語氣。
7.  **作為** 安全稽核員，**我想要** 確信所有 AI 推理模型都在本機運行，**以便** 符合公司嚴格的資安合規政策。
8.  **作為** 用戶，**我想要** 選擇不同大小的 AI 模型，**以便** 在我的輕薄筆電上也能跑得動轉錄功能，而不僅限於工作站。

## 7. 技術要求

### 7.1 後端框架
*   **推薦**：Node.js (Express) 作為控制層，搭配 Python (FastAPI/Sidecar) 作為推理層。
*   **理由**：Node.js 擅長處理 Web UI 請求與檔案系統操作，而 Python 是 AI 模型（Whisper, LLM）的最佳執行環境。兩者透過本地 Socket 或 REST API 溝通。

### 7.2 資料庫
*   **推薦**：SQLite。
*   **理由**：輕量級、無需安裝伺服器、單一檔案形式存儲，完全符合「本地桌面應用」的需求。

### 7.3 資料庫 ORM
*   **推薦**：Prisma (Node.js) 或 SQLAlchemy (Python)。
*   **理由**：提供強型別支援，減少資料庫操作錯誤，且易於維護 Schema。

### 7.4 前端技術
*   **推薦**：React + Tailwind CSS + Electron。
*   **理由**：React 提供豐富的 UI 元件庫；Electron 能將 Web 技術封裝成跨平台的桌面應用程式，並獲得存取本地檔案與麥克風的硬體權限。

### 7.5 其他整合
*   **語音處理**：`Faster-Whisper`（效能優化版 Whisper）。
*   **發言人辨識**：`pyannote-audio`。
*   **本地 LLM**：`llama-cpp-python` (支援 Llama 3 或 Gemma)。

### 7.6 部署環境
*   **封裝方式**：使用 Electron Builder 打包成 .exe (Windows) 或 .dmg (macOS) 安裝檔。
*   **硬體要求**：建議 16GB RAM，支持 NVIDIA GPU (CUDA) 或 Apple Silicon (MPS) 加速推理。

### 7.7 版本控制
*   **推薦**：Git (GitLab 或 GitHub Private Repo)。

## 8. 成功指標

1.  **轉錄準確率 (WER)**：在安靜環境下，繁體中文與英文的字錯誤率 (Word Error Rate) 低於 10%。
2.  **處理效率**：一小時的錄音檔在建議硬體設備上，需在 15 分鐘內完成轉錄與摘要。
3.  **隱私合規性**：通過封包檢測，確保程式在運行過程中無任何未經授權的外連請求。
4.  **用戶滿意度**：MVP 測試用戶針對「摘要可用性」的評分達到 4/5 分以上。
5.  **功能完成度**：MVP 定義的所有 6 大模組需 100% 實作並通過驗收。
6.  **系統穩定性**：連續錄音 4 小時無程式崩潰或資料丟失現象。

## 9. 假設與約束

### 9.1 假設
1.  用戶的電腦具備足夠的磁碟空間存儲錄音檔與 AI 模型（約需 5-10GB）。
2.  用戶具備基礎的電腦操作能力，能自行選取音訊輸入設備。
3.  主要使用場景為近場錄音（麥克風距離說話者 3 公尺內）。
4.  用戶同意在初次啟動時，需要短暫聯網以下載必要的 AI 模型參數（後續可完全離線）。
5.  假設市場上存在對 100% 離線 AI 產品的剛性需求。

### 9.2 約束
1.  **硬體約束**：由於是全離線運行，在效能較差的筆電上，轉錄速度可能慢於錄音時長。
2.  **技術約束**：`pyannote-audio` 模型在處理背景噪音較大的環境時，發言人區分準確度會下降。
3.  **法律約束**：產品必須在界面顯著位置提醒用戶，錄音需符合當地隱私法規（如告知受訪者）。
4.  **交付時程**：必須在 12 週內完成 MVP 的開發與首輪測試。
